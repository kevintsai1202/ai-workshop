## 摘要（TL;DR）

景笙顧問同時操作 **Etsy、Amazon、Shopify、Faire** 四大電商平台，後端用 **NetSuite (ERP)、Hubstaff（工時）、LTS（內部系統）、Google Workspace** 串起跨台美菲三地團隊。本報告盤點：

1. **現有 4 個資訊系統的實際能力範圍** — 哪些功能已經買了但可能沒充分用？
2. **4 個電商平台的商品 / 訂單管理機制** — 每個平台 API 邊界與痛點
3. **與 ERP 整合的 3 條技術路徑** — 原生 SuiteCommerce、第三方 connector、自建 API
4. **景笙當前最有可能的「資料落差」** — 多平台手動對帳、多幣別、跨時區、Ellen 一人三職的瓶頸
5. **短中長期整合策略建議**

**核心觀察**：景笙的 IT 是全外包架構（Tuan / Aivaras / Mobility 為外部供應商，Hannah 為內部主管兼任 PowerBI），這意味著**不適合自建整合，應優先採用 SaaS connector**（如 Celigo / Folio3）；而 AI 自動化（NotebookLM + Skill）是「**connector 落地前**」最划算的補洞工具。

> **本報告與課程規劃的銜接界線**：本報告涉及 connector 選型、API 整合等技術性建議，這些屬於景笙與外包 IT 廠商分軌處理的範圍。**課程本身只聚焦於 10 位內部員工的 AI 應用，不為外包 IT 設計 Skill 場景。**

## Part 2｜景笙四大電商平台的商品與訂單管理

### 2.1 Etsy — Open API v3

**平台定位**：手作 / 個性化商品為主，景笙「Hundred Hearts Personalized」品牌主場

**商品 (Listing) 管理**：

- REST API（OAuth 2.0 認證）
- 可程式化建立 / 更新 / 下架商品
- 支援多變體（variant）、客製化選項（personalization）
- **限制**：標題 140 字元、20 張圖、13 個 tags

**訂單管理**：

- post-purchase 訂單擷取、出貨狀態回報、退款流程
- 個人化訂單細節（buyer 提供的客製化文字）需由 API 取出

**整合特性**：

- ⚠ **V2 API 將於 2026 年底全面退役** — 任何尚在用 V2 的整合需立即遷移到 V3
- v3 OAuth 2.0、JSON 回傳
- 維運成本：版本追蹤、OAuth refresh、合規更新需持續投入

**景笙的可能痛點**：

- 個性化商品的 SKU 變異多 → **庫存追蹤複雜**
- Etsy 不直接提供 SKU 級銷售報表，要靠 API 自行彙整

---

### 2.2 Amazon Seller Central — SP-API

**平台定位**：景笙最大量、最嚴格政策的平台

**商品 Listing 管理（Listings Items API）**：

| 操作 | 用途 |
| --- | --- |
| `putListingsItem` | 建立 SKU + 設定價格 / 庫存 / 屬性 |
| `patchListingsItem` | 部分更新（例如只改價格） |
| `deleteListingsItem` | 下架 SKU |

**庫存管理**：

- 程式化監控庫存等級
- 避免 oversell 與 stockout
- 美倉同事可即時更新可用量

**訂單管理**：

- 即時擷取訂單、確認出貨、處理取消與退款
- 出貨追蹤資訊雙向同步

**支付 / 報表**：

- Settlement Report（結算報表）
- Inventory Reports
- Sales & Traffic Reports

**整合特性**：

- REST + LWA (Login with Amazon) 認證
- Rate limit 嚴格（不同 endpoint 有不同 quota）
- **政策變動頻繁**：A-to-Z Guarantee、FBA 規則、Brand Registry — Camille 客服回覆英文客訴的 grounding 來源就是這些

**景笙的可能痛點**：

- 24 小時內必須回覆客訴 → 菲律賓 CS 夜班輪值
- 多 marketplace（US / EU / UK 各自獨立）→ 每個 marketplace 都要單獨設定

---

### 2.3 Shopify — Admin API（GraphQL 為主）

**平台定位**：景笙的 D2C 主力 / 品牌官網

**商品管理**：

- Product / Variant / InventoryItem / Location 四層結構
- **多倉支援**：每個 SKU 可在多個 Location 有不同庫存

**庫存管理（Inventory API）**：

- 庫存狀態細分：**available / reserved / on-hand / incoming / committed / damaged**
- GraphQL Bulk Update Inventory API：可一次更新數千 SKU 不撞 rate limit
- Fulfillment Orders API：控制每筆訂單由哪個倉出貨

**訂單管理**：

- Order / Transaction / Refund / Fulfillment 四層結構
- Webhook 即時推送訂單變動

**整合特性**：

- **2025 年 4 月起所有新公開 app 必須用 GraphQL**（REST 進入維護模式）
- 2026-04 起 idempotency key 強制使用（透過 `@idempotent` directive）
- 認證：OAuth 2.0 + access token

**景笙的可能優勢**：

- Shopify 是四個平台中**整合生態最成熟**的 → Celigo / Folio3 都有預建 NetSuite connector
- Faire 也有官方 Shopify app，可雙向同步

---

### 2.4 Faire — B2B 批發平台

**平台定位**：B2B 批發，買家是各國獨立精品店（25,000+ 北美零售商）

**商品管理**：

- 透過 Faire Brand Portal 上架
- 批發價、最低訂購量 (MOQ)、首次下單折扣
- 與 Shopify 雙向同步（**官方推薦的整合路徑**）

**訂單管理**：

- 訂單 API：拉取訂單、回傳出貨追蹤
- API key 需向 Faire 團隊申請（不是自助開通）

**整合特性**：

- API key 需「**Generate API Key**」按鈕產生，由 Faire 審核
- 主要透過 OMS 整合：**InfiPlex / Sellercloud / Cin7 Core** 是常見選項
- Shopify 整合是**最簡單路徑** — 「Faire: Import, Export, Fulfill」app 即時雙向同步庫存 / 訂單 / 出貨

**景笙的可能策略**：

- 既然 Shopify 已有 → 走「**Faire ↔ Shopify ↔ NetSuite**」三層架構，比直接做 Faire ↔ NetSuite connector 更省

## Part 3｜電商平台與 NetSuite 的整合方案

### 3.1 三條技術路徑比較

| 路徑 | 優點 | 缺點 | 適合景笙嗎？ |
| --- | --- | --- | --- |
| **A. SuiteCommerce 原生** | 100% 原生整合，無 middleware | 需把店面搬到 SuiteCommerce，捨棄 Shopify | ❌ 景笙已有 Shopify |
| **B. 第三方 Connector**（Celigo / Folio3 等） | 預建好、2-4 週上線、SaaS 訂閱模式 | 月費 / 年費（5-15 萬美金/年區間）、依賴第三方維運 | ✅ **最推薦** |
| **C. 自建 API 整合** | 完全客製、無持續授權費 | 需內部開發 + 維運能量 | ❌ 景笙 IT 全外包，不適合 |

### 3.2 第三方 Connector 主要選項

#### Celigo（市占最高）

- **NetSuite-Shopify Connector**：訂單 / 出貨 / 付款 / 庫存 / 退款雙向同步
- **NetSuite-Amazon Connector**：FBA / FBM 訂單、Settlement Report
- 涵蓋平台：Shopify、Amazon、eBay、Walmart、HubSpot 等 100+
- 建置時間：標準實作 **2-4 週**
- 模式：SaaS 訂閱

#### Folio3

- **直接安裝在 NetSuite SuiteApp**，不需第三方 infra
- 預建 Shopify-NetSuite、WooCommerce-NetSuite、Magento-NetSuite
- 客戶實例：Tone It Up（健身服飾）等
- 模式：一次性費用 + 維運費

#### 其他（簡記）

- **Jitterbit**：通用 iPaaS，彈性高但需配置工
- **InfiPlex**：強在 Faire OMS 整合
- **Sellercloud**：多 marketplace 中央管理

### 3.3 推薦的整合架構（給景笙）

```
                    ┌──────────────────────┐
                    │     NetSuite ERP      │  ← 唯一財務真相來源
                    │  (含財務、庫存、OMS)   │
                    └──────────▲───────────┘
                               │
              ┌────────────────┼────────────────┐
              │                │                │
   ┌──────────┴────┐  ┌────────┴──────┐  ┌─────┴──────┐
   │  Celigo /     │  │  Celigo /     │  │  Celigo /  │
   │  Folio3       │  │  Folio3       │  │  Folio3    │
   │  (Shopify     │  │  (Amazon      │  │  (Etsy v3  │
   │   Connector)  │  │   SP-API)     │  │   Custom)  │
   └──────▲────────┘  └──────▲────────┘  └─────▲──────┘
          │                  │                  │
   ┌──────┴────┐      ┌──────┴────┐     ┌──────┴────┐
   │  Shopify  │◀─────┤  Amazon   │     │   Etsy    │
   │   (D2C)   │      │  (US/EU)  │     │  (手作)    │
   └──────▲────┘      └───────────┘     └───────────┘
          │
   ┌──────┴────┐
   │   Faire   │  ← 走 Shopify app 串
   │  (B2B 批發)│
   └───────────┘
```

**設計重點**：
1. **NetSuite 是唯一 source of truth**，所有財務、庫存最終匯總到這裡
2. **Faire 走 Shopify app 連接**，不直接接 NetSuite，省一條 connector 訂閱
3. **Etsy v3 較少預建 connector**，可能要自建或用 API2Cart 之類聚合服務

**根本解**：NetSuite 多幣別模組已有此功能 — 確認是否啟用
