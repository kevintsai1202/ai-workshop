### 1-4 核心觀念講授：Agent Skills 是什麼？

#### A. 一句話定義
> **Skill = 把你重複用的提示詞 + 流程 + 工具，打包成一個可以一鍵呼叫的單位。**

#### B. Skill 的三要素

| 要素 | 說明 | 例子（發票辨識 Skill） |
| --- | --- | --- |
| **觸發條件**（Trigger） | 什麼時候啟動？ | 我說「整理發票」或拖入發票照片 |
| **輸入**（Input） | 它需要什麼資料？ | 發票照片 1-N 張 |
| **輸出**（Output） | 它會吐出什麼？ | 一張 Markdown 表格，可貼到 Excel |

#### C. Skill 與「直接用 AI」的差異

| 比較項 | 直接用 AI | 用 Skill |
| --- | --- | --- |
| 需要每次貼提示詞 | ✅ 是 | ❌ 不用，已封裝 |
| 同事可以用 | ❌ 提示詞要傳 | ✅ Skill 可分享 |
| 流程一致性 | ❌ 每次可能不同 | ✅ 固定流程 |
| 改善可累積 | ❌ 改一次只在自己對話 | ✅ 改 Skill 大家受惠 |

#### D. 為什麼這對行政人員很重要？
- 你做的 80% 行政工作其實是**有規則的重複**。
- 把規則封裝成 Skill，就是**把自己的經驗變成資產**。
- 你下班後 Skill 還能用——團隊可以重複享用你的工作成果。

### 2-2 範本拆解（30 分鐘）

> 以下用簡化過的 Skill YAML / Markdown 格式說明（依實際 Antigravity / Claude Skill 格式調整）。
>
> 📎 **教學素材**：本範例 Skill 之 prompt 規則，可直接引用 [公司費用報支管理辦法](../教學素材/公司費用報支管理辦法.md)（[PDF](../教學素材/pdf/公司費用報支管理辦法.pdf)）第二章報支標準作為 knowledge source。**講師示範時可以打開 Antigravity，把這份 PDF 拖入 Skill 的 knowledge 設定區**，讓學員看到「規章是怎麼餵給 Skill 的」。完整 Skill 範本見 [課程輔助文件](../課程輔助文件.md) 第 6 點。

#### 範例 Skill 結構

```yaml
---
name: invoice-to-excel
description: |
  把多張發票照片辨識並整理成 Excel 表格。
  使用時機：每月報帳、整理收據時。
trigger:
  - 用戶輸入「整理發票」
  - 用戶拖入發票照片（圖片檔）
inputs:
  - name: invoice_images
    type: image
    required: true
    description: 1-N 張發票照片
outputs:
  - name: excel_table
    type: markdown_table
    description: 含日期、商家、金額、用途的表格
steps:
  - 辨識每張發票的關鍵欄位
  - 統一日期格式為 YYYY-MM-DD
  - 計算總計
  - 輸出 Markdown 表格
prompt: |
  你是會計助理。
  請依以下規則辨識上傳的發票...
  [此處貼上 Day 3 用過的提示詞]
---
```

#### 拆解講解（每欄位都要講）

| 欄位 | 必填？ | 重要性 | 講師說明重點 |
| --- | --- | --- | --- |
| `name` | ✅ | ★★★ | 英文小寫加底線，後續 AI 用這個名字找 Skill |
| `description` | ✅ | ★★★★★ | **最重要**！AI 靠這段判斷「我什麼時候該用這個 Skill」 |
| `trigger` | ⚠️ | ★★★ | 列出觸發場景，越具體越精準 |
| `inputs` | ✅ | ★★ | 定義要的資料 |
| `outputs` | ⚠️ | ★★ | 預期產出 |
| `steps` | ⚠️ | ★ | 文件性質，幫人讀懂 |
| `prompt` | ✅ | ★★★★ | 真正執行的提示詞，等同你前 3 天累積的精華 |

> **講師備忘**：強調 `description` 寫法，這決定 Skill 能不能被自動觸發。

### 2-3 講師示範：用 AI 寫 Skill 初稿（30 分鐘）

#### 示範流程
1. 講師選一個情境：「綠野選物 — 客訴回覆 Skill」
2. 在 Antigravity / Codex 中說：
   > 「請幫我建立一個 Skill 範本，名稱叫 `customer-complaint-reply`，用途是把顧客的負評整理出回覆草稿。我會貼一段 Day 3 用過的提示詞做為主體。」
3. AI 產出初稿。
4. 講師逐欄修：把 description 寫到「AI 看了就知道何時呼叫」。

#### 改 description 的對比示範

**❌ 爛 description：**
> 客訴回覆工具

**✅ 好 description：**
> 將顧客的負面評論（Google 評論、FB、IG 私訊）整理成 150 字內、含同理 + 解決方案 + 邀請的回覆草稿。使用時機：客服每日處理客訴時、店長回覆評論時。

#### 階段 1：補完 Skill（30 分鐘）
- 補上 trigger / outputs / steps 欄位。
- 把 prompt 從 Day 1~3 提示詞庫貼進來。
- 加上 1-2 個 few-shot 範例（讓 Skill 跑起來更穩定）。

#### 階段 2：實測（30 分鐘）
- 用真實資料跑 1 次 Skill。

#### 階段 3：迭代（30 分鐘）
- 根據實測結果，回頭改 description 或 prompt。
- **常見迭代點**：
  - description 不夠具體 → AI 不會自動觸發

## 講師備忘：時間控制與容易卡住的點

| 時段 | 容易卡住的點 | 因應 |
| --- | --- | --- |
| 工具安裝 | 公司筆電權限 | 改用助教筆電 / 共看講師螢幕 |
| Skill 初稿 | description 寫不出來 | 教問句法：「我希望 AI 在 ____ 時候自動用這個」 |
| 實測 | Skill 跑出來結果亂七八糟 | 先檢查 prompt 有沒有「不要做什麼」的限制 |
| 成果展示 | 學員不敢上台 | 講師先指定 2 位作品較好的同學 break ice |
| 後測 | 學員拖延 | 嚴格 30 分鐘倒數計時，到點停筆 |

## 備援素材（萬一進度太快）

- **加碼**：示範把 2 個 Skill 串接（例：辨識發票 → 寄出報帳 Email）。
- **加碼**：教學員把 Skill 加進排程，每天自動跑（例：每天早上自動回覆昨晚 IG 私訊）。
- **加碼討論**：「未來 1 年，你公司可以做多少個 Skill？這些 Skill 應該屬於誰？」
