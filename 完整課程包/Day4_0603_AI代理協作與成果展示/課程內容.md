### 1-4 核心觀念講授：Agent Skills 是什麼？

#### A. 一句話定義（依 [agentskills.io](https://agentskills.io/home) 官方規範）

> **Skill = 一個資料夾，裡面有一份 SKILL.md，告訴 AI「什麼時候用、做什麼、怎麼做」。**
>
> Agent Skills 是 Anthropic 釋出的**開放標準**，目前 30+ 種工具支援（Codex、Claude、Cursor、VS Code、Junie⋯）。學一次，跨工具通用。

#### B. Skill 的核心結構（4 件事）

| 元件 | 必填？ | 規格 | 說明 |
| --- | --- | --- | --- |
| **資料夾** | ✅ | 名稱 = `name` 欄位 | 含 SKILL.md，可選 `scripts/`、`references/`、`assets/` 子目錄 |
| **`name`**（YAML） | ✅ | 1-64 字，**lowercase + hyphen**，不可底線/大寫/連續 hyphen | 必須與資料夾同名 |
| **`description`**（YAML） | ✅ | 1-1024 字，含「**做什麼**」+「**什麼時候用**」 | **最重要**！AI 靠這段判斷觸發 |
| **Markdown body** | ✅ | 建議 < 5000 tokens | 真正的執行指令：步驟、輸出格式、規則、邊界情況 |

> **觸發機制（progressive disclosure）**：AI 啟動時只讀所有 skills 的 name + description（每支約 100 tokens），匹配到才載入完整 SKILL.md body。所以 description 寫得越具體、含關鍵字，AI 越精準觸發。
>
> **常見錯誤**：name 寫成 `auto_invoice_parser`（含底線）、`Invoice-To-Excel`（含大寫）、`pdf--parser`（連續 hyphen）— 都會讓 skill 載不進去。正確寫法：`invoice-to-excel`。

#### C. Skill 與「直接用 AI」的差異

| 比較項 | 直接用 AI | 用 Skill |
| --- | --- | --- |
| 需要每次貼提示詞 | ✅ 是 | ❌ 不用，已封裝 |
| 同事可以用 | ❌ 提示詞要傳 | ✅ Skill 可分享 |
| 流程一致性 | ❌ 每次可能不同 | ✅ 固定流程 |
| 改善可累積 | ❌ 改一次只在自己對話 | ✅ 改 Skill 大家受惠 |

#### D. 為什麼這對行政人員很重要？
- 你做的 80% 行政工作其實是**有規則的重複**。
- 把規則封裝成 Skill，就是**把自己的經驗變成資產**。
- 你下班後 Skill 還能用——團隊可以重複享用你的工作成果。

> **講師備忘（呼應 SPA）**：本段「為什麼對行政重要」可搭配 SPA「Gem → Skill 升級」對比卡（concept #7）一起秀給學員看，讓他們從 Day 1 做的 Gem 直接連到 Day 4 要做的 Skill — 加深「升級感」印象。

### 2-2 在 Antigravity 安裝 Skill：自製與社群兩種來源（20 分鐘）

> 一個 skill 可以**自己寫**，也可以**裝別人寫好的**（從 skills.sh、agentskill.sh 等社群庫）。Antigravity 採 agentskills.io 規範，**啟動時自動發現** skill 目錄裡的所有 SKILL.md。本節先帶大家把社群 skill 裝起來——包括下節 2-4 會用到的 **skill-creator**——之後再看 SKILL.md 內部結構與自己寫一個。

#### Skill 目錄路徑（依 [Antigravity 官網](https://antigravity.google/docs/skills)）

| 範圍 | macOS / Linux | Windows | 何時用 |
| --- | --- | --- | --- |
| **全域**（推薦） | `~/.gemini/antigravity/skills/<name>/` | `%USERPROFILE%\.gemini\antigravity\skills\<name>\` | 所有專案共用（推薦給個人 skill） |
| **工作區** | `<project>/.agent/skills/<name>/` | `<project>\.agent\skills\<name>\` | 僅當前專案，**優先於全域** |

> **⚠ 跨工具路徑差異（容易踩坑）**：
>
> | 工具 | Skill 目錄 |
> | --- | --- |
> | **Antigravity** | `~/.gemini/antigravity/skills/` |
> | **Gemini CLI** | `~/.gemini/skills/`（注意：**沒有** `antigravity` 子目錄！） |
> | **Codex App** | 透過 Plugins UI 裝（不直接放檔案） |
> | **Claude Code** | `.claude/skills/`（專案層）/ `~/.claude/skills/`（全域） |
>
> 同一份 SKILL.md **內容可跨工具通用**，但**放置路徑不同**。如果你既用 Antigravity 又用 Gemini CLI，最聰明的做法是**用 symbolic link 共用**：
>
> ```bash
> # macOS / Linux
> ln -s ~/.gemini/skills ~/.gemini/antigravity/skills
> ```
>
> ```powershell
> # Windows（PowerShell 7+，需管理員權限）
> mklink /D "$env:USERPROFILE\.gemini\antigravity\skills" "$env:USERPROFILE\.gemini\skills"
> ```

> **講師備忘**：學員第一次裝建議放**全域**（`~/.gemini/antigravity/skills/`）— 之後在任何專案都能呼叫。Windows 學員務必確認家目錄路徑（`echo $env:USERPROFILE` 印出來看）。

#### 三種安裝方式（依用途分類）

> - **方法 A 適用：把自己寫好的 skill 放進來**（下節 2-4 寫完會用）
> - **方法 B / C 適用：裝別人寫好的社群 skill**（本節馬上用，當作社群 skill 入門）

**A. 手動拷貝（最直觀，自製 skill 用）**

```bash
# 確認目錄存在
mkdir -p ~/.gemini/antigravity/skills

# 把你的 skill 資料夾整個複製進去（invoice-to-excel 是 2-4 我們會自己寫的，先看流程）
cp -R ./invoice-to-excel ~/.gemini/antigravity/skills/

# 重啟 Antigravity（或開新對話串）
```

**B. `/learn` 命令（從 agentskill.sh 安裝社群 skill）**

```bash
# 一次性註冊 /learn 命令（全域安裝 ags 工具）
git clone https://github.com/agentskill-sh/ags.git ~/.gemini/antigravity/skills/learn
```

```text
# 之後在 Antigravity 對話中：
/learn seo                                ← 搜尋
/learn @anthropic/seo-content-optimizer   ← 安裝特定 skill
/learn trending                           ← 看熱門 skill
```

**C. `npx` 安裝器（批次或單一安裝）**

```bash

npx antigravity-awesome-skills --antigravity

# 單一安裝
npx @rmyndharis/antigravity-skills install <skill-name>
```

#### 講師示範：裝 skill-creator 當熱身（10 分鐘）

> 用 `npx skills add` 從 skills.sh 裝 [skill-creator](https://github.com/anthropics/skills/tree/main/skill-creator) — 它是 Anthropic 官方推出的 meta-skill（用來幫你寫其他 skill）。**先把它裝起來**，下節 2-4 會直接用它寫第一個自製 skill。

```bash

```

重啟 Antigravity 對話後，輸入：

> 「啟動 skill-creator」

學員應該看到 skill-creator 主動向你問 5-7 個問題。**先讓它跑一輪**（可暫時不寫完），體驗「skill 是怎麼觸發的」。下節我們會看示範 skill 的內部結構，再回來用它寫自己的。

#### 驗證安裝

1. **重啟 Antigravity 對話串**（progressive disclosure 在啟動時掃描 skill 目錄）
2. 在對話中輸入 skill 的觸發場景文字（例：「啟動 skill-creator」）
3. Antigravity 應自動辨識對應 skill 並啟動
4. 若沒觸發 → 檢查：
   - 資料夾名是否與 `name` 欄位完全相同？
   - SKILL.md 在資料夾**根層**而不是子目錄？
   - `description` 是否含足夠關鍵字？

> **講師備忘**：
>
> - **跨工具差異**：Codex 用 `/skills` 指令、Claude Code 預設掃 `.claude/skills/`、Antigravity 用 `~/.gemini/antigravity/skills/`。同一份 SKILL.md 內容可跨工具，**只是放置路徑不同**。
> - 學員若要同時在 Codex App 與 Antigravity 用同一個 skill，可用**符號連結**或在兩處各放一份。
> - 第一次裝完務必重啟對話串，不少學員會忘了這步。

### 2-3 範本拆解（25 分鐘）

> 你已經會裝 skill 了，現在我們**打開一個示範 skill 的 SKILL.md**，看看裡面長什麼樣，準備等下動手寫一個。以下範本依 [agentskills.io 官方規範](https://agentskills.io/specification) 撰寫。
>
> 📎 **教學素材（兩份 .md 對比示範）**：本範例 Skill 的合規規則來源備有兩個版本，**講師務必兩份都打開給學員看**，作為「references 該放什麼樣的檔」的活教材：
>
> | 檔案 | 角色 | 用途 |
> | --- | --- | --- |
> | [公司費用報支管理辦法.md](../教學素材/公司費用報支管理辦法.md) | **完整版**（18 條法規體例） | 適合**人讀**、做學員 Skill 跑出結果後的人工驗收 |
> | [expense-policy.md](../教學素材/expense-policy.md) | **Skill 精簡版**（硬規則 + 合規速查表） | 推薦**直接放進 skill `references/`**，AI 取用 token 少、判讀更準 |
>
> **講師示範時打開 Antigravity / Codex App，建立 `invoice-to-excel/` 資料夾並把** `expense-policy.md` **放進 `references/` 子目錄**，讓學員看到「規章是怎麼餵給 Skill 的」；**同時對照打開完整版**，引導學員看「同樣的規章，給人讀 vs 給 AI 讀，要怎麼裁切」。
>
> 💡 **核心觀念**：references/ 是 AI **按需載入**的知識來源，PDF 要先做 OCR / text-extract 解析才能讀、每次觸發多花 token；純文字 .md 可直接 tokenize，省成本也讓抓取更準。**精簡版**進一步把抽象條文改寫成「條件 → 判定 → 動作」速查表，讓 AI 跑「合規檢查」時直接套用。學員若手上只有 PDF，可請 AI 先轉成 .md 再人工抽取硬規則。完整 Skill 範本見 [課程輔助文件](../課程輔助文件.md) 第 6 點。

#### 範例 Skill 結構：資料夾 + SKILL.md

```text
invoice-to-excel/                ← 資料夾名 = name 欄位（必須相同）
├── SKILL.md                     ← 必要：metadata + 指令
```

#### 拆解講解（每欄位都要講）

| 欄位 / 元件 | 必填？ | 重要性 | 講師說明重點 |
| --- | --- | --- | --- |
| **資料夾名** | ✅ | ★★★ | 必須與 `name` 完全相同。檔案結構：`<name>/SKILL.md` |
| `name`（YAML） | ✅ | ★★★ | 1-64 字，**全小寫 + hyphen**。**底線、大寫、連續 hyphen、首尾 hyphen 都會讓 skill 載不進去**。例：`invoice-to-excel`（✅）、`auto_invoice_parser`（❌） |
| `description`（YAML） | ✅ | ★★★★★ | **最重要！** 1-1024 字，必須同時說明「**做什麼**」+「**什麼時候用**」。AI 啟動時只讀這段判斷觸發 |
| `license`、`compatibility`、`metadata`、`allowed-tools`（YAML） | ⚠️ | ★ | 選填。多數 Skill 不需要（除非要對外發布或有特殊環境需求） |
| **Markdown body** | ✅ | ★★★★ | 真正執行的指令：角色、輸出格式、處理規則、邊界情況。建議 < 5000 tokens |
| `references/`（資料夾） | ⚠️ | ★★ | 選填。放補充文件（如公司規章 PDF）給 AI 按需載入 |
| `scripts/`、`assets/`（資料夾） | ⚠️ | ★ | 選填。`scripts/` 放可執行程式、`assets/` 放靜態資源 |

> **講師備忘 1**：強調 `description` 寫法 — 這決定 Skill 能不能被自動觸發。**「做什麼」+「什麼時候用」雙重作用**是官方指引。
>
> **講師備忘 2**：注意官方規範**沒有 `trigger` / `inputs` / `outputs` / `steps` / `prompt` 等欄位**。觸發靠 description、輸入輸出與步驟全寫在 Markdown body 裡。學員若以為這些是必填欄位，會做出無法載入的 skill。

### 2-4 講師示範：寫 Skill 初稿（30 分鐘）

> 看完範本結構，現在動手寫一個你自己的。兩條路徑：**手寫**（讓 AI 隨意產出再修）、**skill-creator**（剛剛 2-2 已經裝好了，直接拿來用）。建議講師示範後者為主，前者當作對照組。

#### 路徑 A. 手寫：直接讓 AI 產出 SKILL.md（10 分鐘）

1. 講師選一個情境：「綠野選物 — 客訴回覆 Skill」
2. 在 Antigravity / Codex 中說：
   > 「請幫我建立一個 SKILL.md（依 agentskills.io 官方規範），name 叫 `customer-complaint-reply`，用途是把顧客的負評整理出回覆草稿。我會貼一段 Day 3 用過的提示詞做為主體。**只填 name + description 必要欄位，指令本體寫在 Markdown body**。」
3. AI 產出初稿。
4. 講師逐項修：把 description 寫到「AI 看了就知道何時呼叫」。

#### 路徑 B. 推薦：用 skill-creator（15 分鐘）

> **skill-creator** 是剛剛 2-2 已經裝好的 meta-skill — 它本身就是 Skill，但專門用來引導你寫**其他** Skill。**與手寫路徑相比，最大差別是它會「問對問題」**：你想清楚才寫，而不是 AI 自由發揮然後你逐欄修。

##### 為什麼推薦它？

| 痛點（手寫常見） | skill-creator 怎麼解 |
| --- | --- |
| 不知道 description 該寫什麼 | 它會問你 5-7 個問題，逼你想清楚「**什麼時候會用**」「**輸出長什麼樣**」 |
| 寫完不知道對不對 | 它會幫你檢查命名、必填欄位、長度限制 |
| description 寫了但 AI 沒觸發 | 它會跑「**觸發測試**」，給優化建議（最多 5 輪迭代） |

##### 使用流程

1. 在 Codex / Claude / 其他 agent 中說：
   > 「啟動 skill-creator，幫我寫一個 Skill。我想做的是『客訴回覆草稿產出』。」
2. **意圖捕捉**：skill-creator 問你 5-7 題（任務、觸發場景、輸入、輸出、邊界情況）
3. **草稿產出**：自動產出符合官方規範的 SKILL.md
4. **驗證測試**（可選）：跑幾個測試提示，看 description 觸發準不準
5. **description 最佳化**（可選）：迭代 5 輪，逼出最精準的 description

> **講師備忘**：「自己寫 Skill 像自己寫程式 — 你會慢慢學會。但用 skill-creator 像有資深前輩坐你旁邊 — **他不會幫你寫，但會問對問題**。」

#### 改 description 的對比示範（5 分鐘）

不論用哪條路徑，都要強調 **description 是 Skill 能不能被觸發的命脈**。

**❌ 爛 description：**
> 客訴回覆工具

**✅ 好 description：**
> 將顧客的負面評論（Google 評論、FB、IG 私訊）整理成 150 字內、含同理 + 解決方案 + 邀請的回覆草稿。使用時機：客服每日處理客訴時、店長回覆評論時。

**為什麼好？** 同時含「**做什麼**」（150 字內、含同理+解決方案+邀請）+「**什麼時候用**」（客服每日、店長回覆評論）。AI 看到使用者貼負評時，會精準匹配到這個 skill。

## Skills 進階探索（15:30 ~ 16:00）

> 上午我們做了「**Skill**」，下午要釐清另一個常被混淆的概念：**插件 (Plugin)**。它們不是同一種東西 — Plugin 是**容器**，Skill 是**元件**，**Plugin 通常包含多個 Skills**。接著以 OpenAI Codex 為例看實際運作，然後帶大家裝 Codex App，最後逛 skills.sh — 一個跨工具 Skills 寶庫。

### A. 插件 vs Skills：誰包含誰（5 分鐘）

| 維度 | 插件（Plugin） | Agent Skills |
| --- | --- | --- |
| **角色** | **容器** — 打包多個功能模組 | **單一元件** — 一個指令模板 |
| **包含內容** | Skills + Apps（外部服務）+ MCP Servers | YAML/Markdown 定義（input / output / description） |
| **例子** | 「GitHub Workflow」、「Slack 自動化」插件 | 「PR 審閱」、「發票整理」、「會議紀錄」 |
| **修改門檻** | 通常需懂程式 | YAML 檔可直接編輯 |
| **使用方式** | **裝來用** | **裝來用 or 自己寫**（今天上午做的事） |

> **講師重點口訣**：「**Plugin 是禮盒，Skill 是禮盒裡的物件**。下一段以 Codex 為例，看 Plugin 內三類元件的實際運作。」

### B. Codex 三件事：Plugin / Skills / Apps（5 分鐘）

> **Codex** 是 OpenAI 推出的 coding agent，有三種介面：
>
> - **Codex App**（桌面）— 多執行緒、Git Worktree 整合、視覺化介面（**下一段我們來裝它**）
> - **Codex CLI**（命令列）— 工程師導向（本課程不涉及）
> - **IDE Extension**（VS Code / JetBrains 外掛）

#### Codex 的 Plugin 內含三類元件

| 名稱 | 是什麼 | Codex 內例子 |
| --- | --- | --- |
| **Plugin** | **容器** — 打包多個元件，可重複使用 | 「GitHub Workflow」插件 |
| **Skills** | Plugin 內的**單一指令模板** | 「PR 自動審閱」、「Issue 摘要」 |
| **Apps** | **外部服務連結**（透過 ChatGPT 帳號驗證） | GitHub、Slack、Google Drive |

> **簡化口訣**：「**Plugin 是禮盒，Skill 是禮盒裡的物件，App 是禮盒外的合作廠商。**」

> **講師備忘**：本段純講概念，不要急著示範安裝 — 先讓學員理解「為什麼 Plugin 比 Skill 大」，下一段一併看安裝流程。

### C. Codex App 安裝 + Plugin 試裝（10 分鐘）

| 實測 | Skill 跑出來結果亂七八糟 | 先檢查 Markdown body 有沒有「不要做什麼」的限制 |
| 插件安裝 | 公司網路擋插件市集連線 | 講師預先在自己機器裝好示範用插件當備援；OpenAI 額度為 0 的學員改示範免費插件 |
| 後測 | 學員拖延 | 嚴格 30 分鐘倒數計時，到點停筆 |

## 備援素材（萬一進度太快）

- **加碼**：示範把 2 個 Skill 串接（例：辨識發票 → 寄出報帳 Email）。
- **加碼**：教學員把 Skill 加進排程，每天自動跑（例：每天早上自動回覆昨晚 IG 私訊）。
- **加碼討論**：「未來 1 年，你公司可以做多少個 Skill？這些 Skill 應該屬於誰？」
